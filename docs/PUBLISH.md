# Publishing Checklist

This is the release punch list for cutting a new `realitycheck` version.

Scope:

- Use this checklist whenever preparing a new Git tag or publishing to PyPI.
- `docs/CHANGELOG.md` is the release notes source of truth â€” add a new topmost
  version section when cutting a release.
- Release metadata (README status line, CITATION.cff, plugin.json version) is
  auto-synced by `make release-metadata` and `make assemble-skills`.

## Versioning

Use semver-style version bumps:

- Patch (`0.3.4`): bug fixes, docs/packaging-only changes, low-risk cleanups.
- Minor (`0.4.0`): new user-facing commands, flags, schema additions, or
  substantial workflow features.
- Major (`1.0.0`): intentionally breaking CLI/schema/workflow changes that need
  explicit upgrade guidance.

## Version Locations

Version is authoritative in `pyproject.toml`. These files are auto-synced by
`make release-metadata` and `make assemble-skills`:

| File | Field | Synced by |
|------|-------|-----------|
| `pyproject.toml` | `version = "X.Y.Z"` | **manual** (source of truth) |
| `README.md` | Status line (`**vX.Y.Z**`) | `make release-metadata` |
| `README.md` | Test count (`NNN tests`) | `make release-metadata` |
| `README.md` | Directory tree test count | `make release-metadata` |
| `CITATION.cff` | `version:` field | `make release-metadata` |
| `integrations/claude/plugin/.claude-plugin/plugin.json` | `"version"` | `make assemble-skills` |

## Prerequisites

- PyPI account with upload permissions for `realitycheck`
- `~/.pypirc` configured with API token (or use `uv publish` with keyring)
- `uv` installed

## Release Punch List

### Pre-flight

- [ ] Start from a clean tree:
      `git status -sb`
- [ ] Sync with remote:
      `git fetch --tags origin && git pull --ff-only`
- [ ] Pick the next version number (patch/minor/major)

### Update version and metadata

- [ ] Update version in `pyproject.toml`:
      `version = "X.Y.Z"`
- [ ] Sync release metadata (README status + test count + CITATION.cff):
      `make release-metadata`
- [ ] Regenerate skills (syncs plugin.json version + regenerates all skill files):
      `make assemble-skills`
- [ ] Verify version consistency:
      `make release-metadata-check && make check-skills`

### Update docs

- [ ] Update `docs/CHANGELOG.md`:
      add a new topmost `## X.Y.Z - YYYY-MM-DD` section with Added/Changed/Fixed/Tested
- [ ] Update README if install steps, CLI behavior, or supported workflows changed
- [ ] Update README Status line summary (version is auto-synced, but the
      one-line description of what changed is manual)
- [ ] Update `docs/` if any documented behavior or guarantees changed
      (SCHEMA.md, WORKFLOWS.md, etc.)
- [ ] If integration skills changed: verify skill content is correct
      (the templates were regenerated by `make assemble-skills` above)

### Validate

- [ ] Run full test suite:
      `REALITYCHECK_EMBED_SKIP=1 uv run pytest -q`
- [ ] Verify test count matches README (should match after `make release-metadata`):
      `REALITYCHECK_EMBED_SKIP=1 uv run pytest -q 2>&1 | tail -1`
- [ ] Optionally validate against a real data project:
      `uv run python scripts/validate.py`
- [ ] Build fresh artifacts:
      `rm -rf dist/ && uv build`
- [ ] Verify package metadata:
      `uvx --from twine twine check dist/*`
- [ ] Smoke-test the built wheel:
      `uvx --isolated --from dist/realitycheck-X.Y.Z-py3-none-any.whl rc-db --help`

### Commit, tag, publish

- [ ] Stage only release files and review:
      `git add pyproject.toml README.md CITATION.cff docs/CHANGELOG.md integrations/`
      `git diff --staged --name-only && git diff --staged`
- [ ] Commit release metadata:
      `git commit -m "release: vX.Y.Z - summary"`
- [ ] Create annotated tag:
      `git tag -a vX.Y.Z -m "vX.Y.Z"`
- [ ] Push commit and tag:
      `git push origin main && git push origin vX.Y.Z`
- [ ] Publish to PyPI:
      `uv publish dist/realitycheck-X.Y.Z-py3-none-any.whl dist/realitycheck-X.Y.Z.tar.gz`
- [ ] Fallback if `uv publish` is not configured:
      `uvx --from twine twine upload dist/realitycheck-X.Y.Z-py3-none-any.whl dist/realitycheck-X.Y.Z.tar.gz`

### Post-publish verification

- [ ] Verify install from PyPI:
      `uvx --refresh --from "realitycheck==X.Y.Z" rc-db --help`
- [ ] Create GitHub Release (extract CHANGELOG section as notes):
      `gh release create vX.Y.Z --title "vX.Y.Z" --notes-file <path> --latest`
- [ ] Verify GitHub Release page and PyPI project page both show new version
- [ ] Confirm clean tree:
      `git status -sb`

## Notes

- Do not publish from a dirty tree.
- Do not reuse an older `dist/` â€” always rebuild for each release.
- Post-publish install checks may need `uvx --refresh` due to index cache lag.
- PyPI does not allow re-uploading the same version â€” bump to next patch if needed.
- Do not add speculative release notes ahead of an actual cut.
- If a release includes breaking behavior, add upgrade notes to both README and CHANGELOG.
- GitHub Releases are a separate overlay on pushed tags â€” `gh release create` is
  needed to populate the Releases page.

## Troubleshooting

### Token Issues

If upload fails with authentication error:
1. Generate new token at https://pypi.org/manage/account/token/
2. Update `~/.pypirc` with new token
3. Ensure token has upload scope for `realitycheck`

### Build Issues

```bash
rm -rf dist/ build/ *.egg-info/
uv build
```
